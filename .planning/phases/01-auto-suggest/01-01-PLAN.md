  <action>
Add a GET /api/admin/diagnostic/:month endpoint (protected with requireAdmin) that:

1. Fetches all roster_assignments for the given month across all three venues (arkbar, hip, love)
2. Fetches all dj_availability records with status='unavailable' for the same month
3. Cross-references: for each assignment, check if the DJ was marked unavailable for that date+slot
4. Handle post-midnight slots (00:00-01:00, 01:00-02:00) by looking up unavailability on the PREVIOUS calendar day (same logic as getUnavailLookupDate in roster.html)
5. Detect partial block assignments: group each DJ's assignments by date+venue, compare against FIXED_TEMPLATE blocks. If template says DJ X has 3 contiguous slots but roster only has 1-2, flag as partial block.

The FIXED_TEMPLATE data must be duplicated server-side for the diagnostic (copy the structure from roster.html lines 1199-1230). Place it as a const near the endpoint.

After building the diagnostic endpoint, analyze the auto-suggest code in roster.html (lines 1268-1335) and document the confirmed root causes. This is a code-reading task -- no code changes.

Confirmed root causes to document in the SUMMARY:

1. **Slot-by-slot assignment without block enforcement (Love Beach + ARKbar)**:
   - Love Beach pass (lines 1287-1293): iterates each slot independently, calls canAssign per slot. If one slot in a DJ's block fails, the others still get assigned, producing partial blocks.
   - ARKbar pass (lines 1305-1311): same pattern -- independent slot iteration.
   - Compare with HIP pass (line 1329): uses `LOCAL_HIP_SLOTS.every(slot => canAssign(...))` which correctly skips the entire block if any slot fails.

2. **Dropdown shows unavailable DJs as selectable**:
   - buildCell() (lines 916-978) categorizes DJs into available/unavailable/notSubmitted groups and renders all three as selectable options. Unavailable DJs get a warning icon but can still be manually assigned.

3. **No logging of assignment decisions**:
   - The auto-suggest function produces no console output explaining why a DJ was assigned or skipped. No way to diagnose failures without reading code.

The SUMMARY should note that the fix (Plan 02) needs to:
- Add `.every()` block enforcement to Love Beach and ARKbar passes (same pattern as HIP)
- Either disable or remove unavailable DJ options from the dropdown
- Add decision logging throughout the auto-suggest function
