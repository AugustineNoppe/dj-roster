  <name>Task 2: Create data migration script</name>
  <files>scripts/migrate-djs-data.js</files>
  <action>
Create `scripts/migrate-djs-data.js` following the established pattern from `scripts/hash-existing-pins.js`.

The script must perform these steps in order:

**Step 0 — Startup guard:**
- require('dotenv').config()
- Fail loudly with process.exit(1) if SUPABASE_URL or SUPABASE_SERVICE_KEY is missing
- Support a `--force` flag (process.argv.includes('--force')) to allow re-running

**Step 1 — Idempotency check:**
- Count rows in `djs` table. If count > 0 and --force not passed, print count and exit 0
- If --force is passed, DELETE all rows from djs before proceeding (log the action)

**Step 2 — Read legacy tables:**
- Fetch ALL rows from `dj_rates` (columns: name, rate)
- Fetch ALL rows from `dj_pins` (columns: name, pin)
- Print row counts for both tables. If either returns 0 rows, print ERROR and exit 1

**Step 3 — Deduplicate and audit (critical section per STATE.md blocker):**
- Create a `canonicalizeName(name)` function: `name.trim().replace(/[\u2013\u2014]/g, '-')` to normalize en-dash/em-dash to hyphen for keying only
- Build a Map from dj_rates keyed by `canonicalizeName(name).toLowerCase()`
- On duplicate key: log WARNING with both name variants, keep FIRST occurrence
- Build a Map from dj_pins keyed by `canonicalizeName(name).toLowerCase()`
- Log any names that exist in dj_rates but NOT in dj_pins (WARNING, will skip these)
- Log any names that exist in dj_pins but NOT in dj_rates (WARNING, informational)

**Step 4 — Build insert rows:**
- For each entry in rateMap, find matching pin in pinsMap
- Skip if no PIN found (already logged in Step 3)
- Validate pin_hash starts with $2b$ or $2a$ — if not, log ERROR and skip that DJ
- Determine type: if name is in RESIDENTS array ['Alex RedWhite', 'Raffo DJ', 'Sound Bogie'] then 'resident', else 'casual'
- Build the insert object: { name: dj.name (original display name, NOT canonicalized), pin_hash, rate: dj.rate, type, active: true, venues: [], recurring_availability: {}, fixed_schedules: {}, failed_attempts: 0, locked_until: null }

**Step 5 — Insert into djs:**
- Use supabase.from('djs').insert(rows) for batch insert
- If insert error, print error and exit 1
- Print count of inserted rows

**Step 6 — Seed JSONB columns (DB-03):**
- Copy the FIXED_AVAILABILITY constant EXACTLY from server.js lines 133-154 (including the _A12, _SB, _AW, _MT, _MS slot arrays with en-dash U+2013 characters)
- Copy the FIXED_SCHEDULES constant EXACTLY from lib/business-logic.js lines 47-60
- For each DJ name in FIXED_AVAILABILITY: update djs SET recurring_availability = the JS object (NOT JSON.stringify — pass raw object to avoid double-encoding pitfall)
- For 'Davoted': update djs SET fixed_schedules = the FIXED_SCHEDULES['Davoted'] JS object
- Log each update with DJ name and whether recurring/fixed was set
- If any update fails, log error but continue (don't abort)

**Step 7 — Verification output:**
- Fetch all rows from djs with all columns
- For each DJ, print: name, rate, type, active, whether recurring_availability has keys, whether fixed_schedules has keys
- Cross-table check: for each DJ name, query dj_availability to confirm at least one row matches (ilike). Log WARNING if zero matches found
- Print total count and "MIGRATION COMPLETE" banner
- Print next-step instructions: "Verify output above, test DJ logins, then run scripts/drop-legacy-tables.sql"
- Exit with code 1 if any errors occurred during the run, 0 otherwise
