### Integration Issues Found

#### INT-01: CORS Allow-Methods (Low Severity)

**File:** server.js:50 — `Access-Control-Allow-Methods: 'GET, POST, OPTIONS'`
**Missing:** PATCH and DELETE not listed
**Impact:** Only affects cross-origin deployments. Current architecture serves roster.html from the same Express server via `express.static`, so browser requests are same-origin and skip CORS preflight. **No production impact currently.**
**Affected requirements:** ADMIN-03, ADMIN-04, ADMIN-05, ADMIN-07, SCHED-02, SCHED-04 (all satisfied in same-origin; would break in cross-origin)
**Fix:** Add `PATCH, DELETE` to the Allow-Methods header

#### INT-02: Missing `id` in Auth Selects (Low Severity)

**File:** server.js:326 and server.js:761
**Issue:** `checkLockout(djRow)` uses `.eq('id', djRow.id)` to auto-clear expired locks, but both `requireDJAuth` and `/api/dj/login` omit `id` from the select string. `djRow.id` is `undefined`.
**Impact:** Expired lockouts are not auto-cleaned from the DB. Auth still works correctly (checkLockout returns false for expired locks). DJs with previous lockouts accumulate stale `failed_attempts` which resets the counter slower than expected on next lockout cycle.
**Affected requirements:** STAB-01, STAB-02 (both still satisfied — lockout persists and works; only auto-cleanup of expired locks is broken)
**Fix:** Add `id` to both select strings
