<objective>
Audit and fix the availability save path in server.js.

Purpose: POST /api/dj/availability (line 1011) applies `slot.replace(/–/g, '-')` before writing
to Supabase, storing slot values with ASCII hyphens. The GET reader applies normalizeSlot() which
converts hyphens back to en-dashes, so single-session roundtrips look correct. However, if any
existing DB rows were written with en-dashes (from any other code path or direct insert), a
subsequent upsert with hyphens will not match the uniqueness constraint (name, date, slot) and
will INSERT a duplicate row silently. Two rows then exist for the same slot — behavior on read
becomes non-deterministic. The fix is normalizeSlot(slot) on save, matching the convention used
in every other write path in the codebase (assign, batch-assign, signoff, signoff-batch).

Output: server.js with corrected slot normalization on save. No schema changes needed.
</objective>

normalizeSlot (line 79):
```javascript
const normalizeSlot = s => s ? s.replace(/[-\u2013\u2014]/g, '\u2013') : s;
```
Converts any hyphen/dash variant to en-dash (\u2013). This is the canonical slot format used in
all read paths (lines 255, 316, 422, 953) and all write paths EXCEPT the bug at line 1011.

The bug (line 1011 — inside POST /api/dj/availability):
```javascript
const newRows = slots.map(({ date, slot, status }) => ({
  name: name.trim(), date, slot: slot.replace(/–/g, '-'), month, status,
}));
```
`/–/` matches only \u2013 (en-dash) and replaces with ASCII hyphen. Result: `'14:00-15:00'`.
Upsert onConflict key is `'name,date,slot'` — if DB has `'14:00\u201315:00'` (en-dash from
any other insert), this upsert produces a second row with hyphen instead of updating the first.

Correct pattern used by all other write paths:
- roster assign (line 784): `slot: normalizeSlot(slot)`
- roster batch (line 809): `slot: normalizeSlot(slot)`
- signoff (line 1134): `slot: normalizeSlot(slot)`
- signoff-batch (line 1152): `slot: normalizeSlot(slot)`
- unsignoff-day (line 1180): `{ date, slot, ... }` (slot already extracted from normalized key)
