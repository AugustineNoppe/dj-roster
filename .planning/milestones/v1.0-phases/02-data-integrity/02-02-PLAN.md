<objective>
Audit and fix the sign-off flow end-to-end in server.js.

Purpose: The sign-off system uses an append-only log in dj_signoffs where "last action wins" per
DJ+date+slot+venue key. Four endpoints read from this log and compute the net state by iterating
rows into a `latest` map: GET /api/dj/signoffs/:name/:month (line 1191), POST /api/dj/unsignoff-day
(line 1163), GET /api/signoffs/:month (line 1212), and POST /api/roster/finalize (line 1272).

None of these queries specify `.order('timestamp', { ascending: true })`. Supabase returns rows
in default order (primary key / insertion order), which is usually chronological but is not
guaranteed by the API contract. If rows come back in any other order, a later `sign` could be
overwritten in the map by an earlier `unsign`, producing incorrect net state. Under rapid
toggles this is the most likely failure mode.

Secondary checks: verify batch sign-off (POST /api/dj/signoff-batch) inserts the correct rows
and that its error path is surfaced (not silently dropped). Verify unsignoff-day reads only
rows for the correct DJ+date+month before computing net state.

Output: server.js with .order('timestamp', { ascending: true }) on all four signoff read
queries. No schema changes needed (timestamp column already exists at line 1134).
</objective>
